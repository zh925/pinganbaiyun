# 验收标准

以下标准以用户确认后的 PR commit 为最终基线。涉及真实门禁的用例只使用获授权测试凭据，证据必须脱敏。

| ID | Given | When | Then | 优先级 | 验证方式 | 结果 |
|---|---|---|---|---|---|---|
| AC-01 | 全新安装且本地没有门禁 | 用户打开门禁列表 | 显示空状态和新增入口；不扫描、不连接、不请求网络 | Must | 仪器测试+网络观察 | 待验证 |
| AC-02 | 用户填写合法显示名、MAC 和 16 位 hex key | 点击保存 | 生成稳定 id，规范化 MAC/key 后加密落盘，列表显示且 key 掩码 | Must | 单元+仪器测试 | 待验证 |
| AC-03 | 表单任一必填项为空、MAC 非法、key 非 16 位或非 hex | 点击保存 | 对应字段显示明确错误，不落盘、不发起 BLE | Must | 参数化仪器测试 | 待验证 |
| AC-04 | 已存在规范化后相同 MAC 的门禁 | 用户新增另一条相同 MAC | 提示重复并定位已有项，不生成重复配置 | Must | 仪器测试 | 待验证 |
| AC-05 | 已保存门禁 A 且 A 是默认 | 用户编辑显示名或合法 key 并保存 | A 的稳定 id 不变，默认关系仍指向 A，重启后新数据存在 | Must | 单元+仪器测试 | 待验证 |
| AC-06 | 已保存非默认门禁 A | 用户确认删除 | A 被删除，其他门禁和默认关系不变 | Must | 单元+仪器测试 | 待验证 |
| AC-07 | 默认门禁 A 存在 | 用户选择删除 A | 确认文案明确“同时取消默认”；确认后 A 与 defaultId 原子清除，不自动改选 B | Must | 单元+仪器测试 | 待验证 |
| AC-08 | 默认门禁为空且 A 存在 | 用户确认“设为默认” | defaultId 指向 A，列表只有 A 显示默认，重启后保持 | Must | 单元+仪器测试 | 待验证 |
| AC-09 | A 已是默认且 B 存在 | 用户确认把 B 设为默认 | 一次操作把默认从 A 替换为 B，任何可观察时刻最多一个默认 | Must | 并发单元+仪器测试 | 待验证 |
| AC-10 | A 是默认 | 用户确认取消默认 | defaultId 为空；后续冷启动不自动创建任务 | Must | 仪器测试 | 待验证 |
| AC-11 | 存储含悬空 defaultId | APP 读取数据 | 安全清空悬空引用，不崩溃、不自动选择第一项 | Must | 单元测试 | 待验证 |
| AC-12 | 存储密文/JSON 损坏 | APP 读取门禁列表 | 不崩溃、不静默覆盖原数据；展示安全重置提示且需二次确认 | Must | 故障注入仪器测试 | 待验证 |
| AC-13 | 有效门禁 A、蓝牙开启且权限完备 | 用户点击 A 的“开门” | 创建唯一 taskId，展示准确目标和阶段，按确认协议执行 | Must | 真机仪器+门禁测试 | 待验证 |
| AC-14 | 采用 SafeBaiyun 直连方案 | 用户对 A 发起开门 | APP 不执行 BLE scan，不要求 bluetoothName，按已知 MAC 直连 Service `14839ac4-...` | Must | 代码审查+真机日志 | 待验证 |
| AC-15 | 目标 Service 含 read/write/notify/indicate 特征 | 服务发现成功 | APP 按确定性规则选择通道、串行启用 descriptor，确认回调成功后才读取 seed | Must | 假 GATT+真机测试 | 待验证 |
| AC-16 | 使用与 SafeBaiyun 相同的 MAC、key、seed 测试向量 | 构造开门指令 | 生成 20 字节 `A5..5A` 帧，MAC 头、DES 密文和校验逐字节与固定 commit 对照结果一致 | Must | JVM 黄金测试 | 待验证 |
| AC-17 | seed 为空、长度不支持或 DES 构造失败 | 流程进入协议阶段 | 到达协议失败终态，不发送空帧/部分帧，错误信息不泄密 | Must | 参数化单元+假 GATT | 待验证 |
| AC-18 | GATT 写调用成功发起且写回调为 GATT_SUCCESS | 流程完成发送 | 终态文案为“开门指令已发送，请确认门禁状态”，不宣称物理门已开 | Must | 假 GATT+文案检查 | 待验证 |
| AC-19 | GATT API 返回未发起或任一回调 status 非成功 | 对应异步步骤执行 | 流程在有限时间内失败，释放 GATT，提供可理解原因和显式重试 | Must | 故障注入 | 待验证 |
| AC-20 | 已有门禁 A 的任务处于任一非终态 | 用户点击 B、默认触发或 NFC 触发到来 | 不创建第二 GATT 任务、不重复下发；展示 A 正在进行 | Must | 并发仪器测试 | 待验证 |
| AC-21 | 任务处于连接/读取/发送前阶段 | 用户点击取消 | 任务唯一终态为 Cancelled，计时器/GATT 清理，按钮恢复可操作 | Must | 假 GATT+仪器测试 | 待验证 |
| AC-22 | 用户取消后设备返回迟到成功回调 | 回调到达 | UI 和结果保持 Cancelled，不改写成功、不触发下一任务 | Must | 确定性并发单元测试 | 待验证 |
| AC-23 | 连接、服务发现、读 seed 或写入阶段不返回 | 阶段超时 | 在配置的有限时限内唯一终态为 TimedOut，资源释放，无无限重试 | Must | 虚拟时钟单元+仪器测试 | 待验证 |
| AC-24 | 前一次任务已失败/超时/取消 | 用户点击显式重试 | 使用门禁最新保存快照创建新 taskId，仍满足全局单任务约束 | Must | 单元+仪器测试 | 待验证 |
| AC-25 | 自动或手动流程失败 | 用户不操作 | APP 不自动无限重试、不后台循环扫描/连接 | Must | 长时仪器观察 | 待验证 |
| AC-26 | 有合法默认门禁 A、进程不存在且无 NFC intent | 用户从 Launcher 冷启动 | 本次 cold-start token 最多创建一个 A 的自动任务，无二次确认 | Must | 冷启动仪器测试 | 待验证 |
| AC-27 | 没有默认、默认引用失效或默认配置非法 | 用户冷启动 | 不创建 BLE 任务；给出无打扰空态或可操作配置提示 | Must | 参数化仪器测试 | 待验证 |
| AC-28 | 冷启动自动任务因权限/蓝牙未就绪而挂起 | 用户授权或开启蓝牙返回 | 只继续原 taskId；Activity onResume 不创建第二任务 | Must | 真机仪器测试 | 待验证 |
| AC-29 | 进程仍在且 APP 从后台回前台、热启动、旋转或 Activity 重建 | 生命周期事件发生 | 恢复当前 UI/任务，不创建新的默认门禁任务 | Must | 生命周期仪器测试 | 待验证 |
| AC-30 | 默认自动任务正在执行 | 用户编辑或删除对应门禁 | 当前任务使用启动快照且保持唯一；后续重试读取最新状态，删除后不再自动触发 | Must | 并发仪器测试 | 待验证 |
| AC-31 | 设备不支持 BLE或蓝牙关闭 | 用户手动/自动开门 | 不发起 GATT；展示设备不支持或开启引导；拒绝后任务终止 | Must | 真机测试 | 待验证 |
| AC-32 | API 31+ 未授予附近设备权限 | 用户发起开门 | 请求连接方案所需最小权限；拒绝/永久拒绝分别提供终止或设置入口，不循环弹窗 | Must | API 31/33/35 真机 | 待验证 |
| AC-33 | API 26–30 且采用已知 MAC 直连 | 用户发起开门 | 不因本需求请求位置权限或要求定位开关，使用 legacy Bluetooth 权限连接 | Must | API 26/28/30 真机 | 待验证 |
| AC-34 | 飞行模式、Wi-Fi/移动数据关闭，蓝牙按系统允许开启，已保存 A | 用户手动开 A | 完整链路不调用后端/网络并到达可验证终态 | Must | 真机+网络观察 | 待验证 |
| AC-35 | 与 AC-34 相同且 A 是默认，进程不存在 | 用户冷启动 | 离线自动路径最多执行一次，并到达可验证终态 | Must | 真机+网络观察 | 待验证 |
| AC-36 | release 构建及代表性 debug 诊断 | 执行 CRUD、成功和全部失败路径 | UI、日志、Toast、异常、剪贴板、分享中无完整 key、seed 或可重放帧 | Must | 自动敏感串扫描+人工检查 | 待验证 |
| AC-37 | 配置已经加密落盘 | 检查应用私有存储和系统备份行为 | key 不以明文出现，应用备份被禁用，卸载后数据不可由 APP 恢复 | Must | 安全检查+真机备份测试 | 待验证 |
| AC-38 | 当前工程基线 | 运行 `testDebugUnitTest` 与 `assembleDebug` | 两项均成功，且 SafeBaiyun 黄金、存储、调度、并发和错误用例全部通过 | Must | 手动 Gradle 检查 | 待验证 |
| AC-39 | API 26、28、30、31/32、33、34、35 及目标门禁测试矩阵 | 执行核心成功/失败回归 | 每项有设备、APP commit/build、结果和脱敏证据；无未解释崩溃或永久挂起 | Must | 真机兼容测试 | 待验证 |
| AC-40 | 需求 PR 待确认 | 用户评审需求 | 只有 PR URL、commit SHA 和明确“通过”齐全时，开发阶段才可从 backlog 推进 | Must | 流程检查 | 待验证 |

## 非功能验收

| ID | 指标 | 目标 | 验证方式 | 结果 |
|---|---|---|---|---|
| NFR-01 | 并发安全 | 任一进程内同时活跃 BLE 开门任务 ≤ 1 | 并发/生命周期故障注入 | 待验证 |
| NFR-02 | 终态一致性 | 每个 taskId 恰有一个终态，终态后回调不改写结果 | 状态机模型测试 | 待验证 |
| NFR-03 | 有限等待 | 每阶段与全流程均有上限；无无限扫描、连接或重试 | 虚拟时钟+长时观察 | 待验证 |
| NFR-04 | 敏感信息 | key、完整 seed、完整可重放帧泄漏数为 0 | 静态扫描+运行时日志检查 | 待验证 |
| NFR-05 | 离线性 | CRUD、手动和默认开门路径网络请求数为 0 | 飞行模式+流量观察 | 待验证 |
| NFR-06 | 数据持久性 | 正常重启/进程回收后配置和 defaultId 100% 恢复 | 重启/进程回收仪器测试 | 待验证 |
| NFR-07 | 数据故障安全 | 解密/解析失败不崩溃、不静默覆盖原数据 | 存储故障注入 | 待验证 |
| NFR-08 | 主线程响应 | BLE、加密和存储长操作不造成可感知 ANR | StrictMode/性能观察 | 待验证 |
| NFR-09 | 可访问性 | 状态不只依赖颜色，关键控件有可读文本/焦点 | TalkBack+人工检查 | 待验证 |
| NFR-10 | 可追溯性 | 协议黄金向量绑定 SafeBaiyun commit 和源文件行 | 测试/文档审查 | 待验证 |
| NFR-11 | 资源释放 | 成功、失败、取消、超时、销毁后无遗留 GATT/计时器 | 泄漏检查+重复回归 | 待验证 |
| NFR-12 | 兼容性 | API 26–35 矩阵核心路径有明确 PASS/FAIL 证据 | 真机报告 | 待验证 |

## 明确不验收的范围

- 不验收 NFC 读写卡、桌面快捷方式、小组件、云端接口或账号系统。
- 不验收热启动、后台回前台、定时或位置触发的自动开门。
- 在没有设备业务回执证据时，不以“物理门已打开”作为 APP 可证明的验收结果。
- 不验收从第三方 APP/数据库获取 MAC 或 key 的能力。
- 不验收 SafeBaiyun 指定 commit 未证明的第二 Service、广播名匹配、额外命令或 `00/02` 回执。
- 不以模拟器、编译成功或旧测试通过替代授权真实门禁的 GATT 兼容验证。
- 本 Stage 不验收 Android 实现、QA、UAT、生产发布或发布后验证。
