# ADR-002：默认门禁自动开门的触发边界与防重复

- 状态：提议
- 日期：2026-07-17
- 决策人：待用户确认
- 关联 Issue/PR：RUO-4、RUO-5、RUO-7；最终需求 PR 待创建

## 背景

“进入 APP”存在进程冷启动、热启动、后台回前台、Activity 重建、权限页返回等多种解释。自动开门会产生物理影响；宽泛监听 `onResume` 容易因系统弹窗、旋转或导航重复下发。父需求初步验收只明确“冷启动”。

## 选项

1. 仅进程级 Launcher 冷启动自动一次；其他生命周期事件不自动触发。
2. 冷启动、热启动和回前台都自动，并增加后台时长阈值与 30 秒冷却。
3. 每次自动前二次确认；安全但不满足“进入 APP 后直接调用”的少操作目标。

## 决策

提议选择选项 1：进程原本不存在、由 Launcher 正常启动且存在有效默认门禁时，消费一次 cold-start token 并最多创建一个任务。最终决定留给用户。

## 理由

- 满足父需求明确写出的冷启动场景。
- 避免权限页返回、Activity 重建和前后台切换造成意外物理操作。
- token + 全局单任务锁比时间冷却更可确定、更易测试。
- 用户随时可在列表显式手动开门或重试。

## 影响与后续

- 默认流程不监听普通 `onResume` 作为新任务触发器。
- 权限/蓝牙引导返回只继续原挂起 taskId。
- 手动和默认任务竞争同一进程级协调器；后到者不创建 GATT。
- 自动失败默认不自动重试；用户显式重试创建新 taskId。
- 若用户选择选项 2，需另行确认后台最短时长、冷却窗口、成功判定和进程重启处理，并扩充验收矩阵。
