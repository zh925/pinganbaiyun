# ADR-001：RUO-4 协议基线与设备连接方式

- 状态：提议
- 日期：2026-07-17
- 决策人：待用户确认
- 关联 Issue/PR：RUO-4、RUO-5；需求 PR 待创建

## 背景

父需求指定 `dogproton/SafeBaiyun` 实际代码为协议来源。审计 commit `c4ff50888ef9244870a30304b3a5e6d0d5a96eac` 只实现 MAC+key、已知 MAC 直连、单 Service `14839ac4-...`、属性选特征、读 seed、DES 20 字节帧和写回调。

当前 `pinganbaiyun@24032ce` 基于另一仓库实现扫描、广播名、两个 Service、通知组包、附加命令和 `00/02` 回执。两者不能在未经选择时同时作为必选协议。

## 选项

1. 以 SafeBaiyun 最小链路为唯一 RUO-4 基线：已知 MAC 直连，不扫描；只承诺“指令已发送”。
2. 保留当前扩展协议为基线：需用户明确改变/扩展指定来源，并提供真实设备和代码级证据。
3. 双协议运行时回退：增加不可控写入和误开风险，测试矩阵翻倍，不建议在本需求采用。

## 决策

提议选择选项 1。最终决定留给用户；在明确确认前，开发不得选路或合并两套协议。

## 理由

- 与父需求指定来源一致，代码可追溯。
- 已知 MAC 直连减少扫描、定位权限、耗电和身份匹配误判。
- 最小化未知命令与未知 Service 写入，降低物理副作用。
- 对 GATT 写成功使用中性文案，避免把传输层完成误报为物理开门。

## 影响与后续

- 必填协议字段为 MAC、key；显示名为本地业务字段，广播名不是基线必填。
- 需为 SafeBaiyun 20 字节帧建立固定 commit 黄金测试。
- 当前扫描、第二 Service、额外命令和回执只能作为隔离的兼容候选，不得静默回退。
- 需要授权真机确认目标门禁的 seed 长度、Service/Characteristic 组合和写入效果。
- 若用户选选项 2，必须修订 PRD、权限矩阵、验收标准与安全分析后重新确认版本。
