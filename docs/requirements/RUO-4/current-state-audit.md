## 从零实现差距

| 能力 | 需求输入 | 开发阶段必须补齐 |
|---|---|---|
| 项目基线 | Android API 26–35 的提议范围 | 创建可编译工程，固定 Gradle/JDK/SDK 版本并记录手动构建证据 |
| 门禁模型 | 稳定 id、显示名、MAC、16 位 hex key、独立 defaultId | 校验、规范化、唯一 MAC、原子 CRUD 与默认关系 |
| 本地安全 | Keystore 支持的加密存储、禁止备份、全面脱敏 | 存储实现、密钥失效/数据损坏处理与安全测试 |
| 手动开门 | 任一门禁可触发，准确展示阶段 | 进程级协调器、任务快照、唯一终态、取消与显式重试 |
| 自动开门 | 仅 Launcher 进程冷启动自动一次 | cold-start token、全局单任务锁、权限/蓝牙返回续接与生命周期恢复 |
| BLE/GATT | 已知 MAC 直连、单 Service、读 seed 后写帧 | 串行异步操作、API 版本适配、超时、错误映射与资源释放 |
| 协议 | SafeBaiyun 固定 commit 的 20 字节 DES 帧 | 黄金向量、输入边界、失败不发送与设备真机验证 |
| 质量 | 40 条功能验收与 12 条非功能验收 | 单元、假 GATT、仪器、离线、兼容性与授权门禁测试 |

## 不得继承的假设

- 不假设存在 NFC、BLE 扫描、广播名匹配、第二 Service、通知回执或额外命令。
- 不假设存在旧 `DoorConfig`、`SharedPreferences`、数据库或迁移来源。
- 不把 GATT 写回调成功表述为物理门已打开。
- 不把原型、静态检查或编译成功替代真实 Android 生命周期和授权门禁验证。
- 不把未配置的 CI/CD 检查解释为通过。

## 进入开发前置条件

1. 最终需求产物 PR、不可变 head SHA 与完全匹配的人工批准口令齐全。
2. PR 按门禁卡动作合并，并确认 GitHub merged 与需求基线 commit。
3. 三份 ADR 的选择随最终需求版本一并确认。
4. 开发 Issue 仍保持 `backlog`，由产品负责人仅在门禁完成后推进。
5. 真机验证另需授权测试门禁、测试凭据与不进入仓库/Issue 的 Secret 管理方式。
