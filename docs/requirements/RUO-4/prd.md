# PRD：蓝牙门禁离线维护与默认门禁自动开门

## 追踪信息

- 父 Issue：RUO-4
- 负责人：需求组 / 业务分析师（RUO-7）
- 状态：内部评审
- 确认 PR：待本轮最终需求产物 PR 创建
- 确认 commit：待用户确认

## 背景与目标

用户当前每次开门依赖后端取得蓝牙门禁信息，并需多次手动操作。目标是在 Android APP 本地维护门禁资料，使已保存门禁在完全无网络条件下仍能发起 BLE 开门；用户设置唯一默认门禁后，真正冷启动 APP 时自动且仅自动尝试一次。

成功不等同于“物理门已打开”。指定参考实现 `SafeBaiyun@c4ff50888ef9244870a30304b3a5e6d0d5a96eac` 只能证明已知 MAC 直连、读取 seed、构造 DES 帧和 GATT 写入链路，不能证明业务回执。除非设备协议证据补齐，产品成功态统一为“开门指令已发送，请确认门禁状态”。

业务目标：

- 取消门禁资料对后端接口的运行时依赖。
- 将门禁 CRUD、默认关系和开门操作收敛为本地、可恢复、可测试的流程。
- 将自动开门的物理副作用限制在明确的冷启动边界内，防止重复或并发下发。
- 形成与指定代码 commit 可追溯的协议、权限、兼容性和安全基线。

## 用户与场景

| 用户/参与者 | 目标 | 核心场景 |
|---|---|---|
| 门禁使用者 | 少操作、离线开门、理解当前进度和失败原因 | 冷启动自动开默认门、从列表手动开任一门 |
| 门禁配置者 | 安全维护多个门禁配置 | 新增、查看、编辑、删除、设置/取消默认 |
| Android 系统 | 管理蓝牙、权限与生命周期 | 权限弹窗、蓝牙开关、进程/Activity 重建、回调调度 |
| BLE 门禁 | 接收 GATT 指令 | 连接、服务/特征发现、seed 读取、指令写入 |

主场景：

1. 用户新增合法门禁并保存，列表可离线恢复。
2. 用户从列表选择任一门禁手动开门。
3. 用户把某门禁设为默认；下次进程级冷启动时 APP 自动尝试一次。
4. 用户取消默认或删除默认门禁后，后续冷启动不自动开门。
5. 权限、蓝牙、设备或协议不满足时，流程安全终止并提供显式恢复操作。

## 范围

### 包含

- 多门禁本地列表：新增、查看、编辑、删除。
- 唯一默认门禁：设置、更换、取消；删除默认项时同步清空，不自动改选。
- 任一门禁的手动开门。
- 默认门禁的进程级冷启动自动开门；自动触发不二次确认。
- 全局单任务、冷启动幂等 token、终态隔离、取消和显式重试。
- 蓝牙关闭、权限缺失/拒绝、非法配置、连接/服务/特征/读取/写入失败、超时、断连、迟到回调等失败路径。
- 无网络验收、Android 8–15 权限和 API 兼容矩阵。
- 本地密钥保护、日志/UI 脱敏、备份策略和遗留 DES 风险说明。
- 以全新 Android 项目为起点的实现约束；不继承已删除仓库中的旧实现或协议假设。

### 不包含

- 后端门禁接口、账号、云同步、远程开门或联网鉴权。
- NFC 读写、桌面快捷方式、小组件、门禁分享或批量导入导出。
- 自动轮询、后台常驻、地理围栏、定时或后台自动开门。
- 门禁固件修改、设备端安全升级或对物理开门成功的无证据承诺。
- 获取/破解第三方 APP 数据或密钥。
- 采用 `SafeBaiyun` 未证明的第二 Service、广播名匹配、通信密钥/时间同步指令、`00/02` 回执；除非 ADR-001 经用户改选并提供证据。
- 本 Stage 的 Android 代码修复、原型实现、QA、UAT 或发布。

## 业务流程与规则

### 门禁资料流程

1. 用户进入门禁列表；无数据时展示空状态和“新增门禁”。
2. 新增/编辑表单至少包含显示名、MAC、key。`bluetoothName` 仅作为现有数据兼容字段，不属于指定协议必填项。
3. 保存前规范化显示名和大小写，执行全字段校验；校验失败不落盘。
4. 新增成功生成稳定 `id`；编辑保持 `id` 不变，默认关系随该 `id` 保持。
5. 同一规范化 MAC 默认不得重复保存；用户若确需同 MAC 多 key，需另行确认业务场景。
6. 删除始终二次确认。删除默认项时文案明确会同时取消默认；确认后原子地删除配置并清空 `defaultId`，不得自动选择其他门禁。
7. 任何时刻 `defaultId` 为 null 或仅引用一个现存配置；不存在引用必须在读取时安全清空。

### 手动开门流程

1. 用户在列表点击“开门”。
2. 若已有开门任务，拒绝新任务并展示当前门禁和阶段；不得并发连接或下发。
3. APP 校验配置、BLE 能力、蓝牙开关和所需权限。
4. 按 ADR-001 确认的连接方案执行定位/连接、服务发现、通道准备、seed 读取、DES 帧构造和写入。
5. 页面连续展示准备、连接、读取、发送和终态；不存在扫描时不得伪造“扫描中”。
6. 用户可在非终态取消。取消后清理超时、GATT 和挂起操作；迟到回调不得改为成功。
7. 失败不无限重试。默认无自动重试；用户显式点击“重试”创建新任务。若后续决定允许一次自动重试，必须限定错误类别并经用户确认。

### 默认门禁与自动开门

1. 默认最多一个，也可为空；设置另一门禁时原子替换。
2. 本稿将“进入 APP”定义为：进程原本不存在，用户通过 Launcher 冷启动并进入主 Activity。该定义待 ADR-002 确认。
3. 热启动、后台回前台、Activity/Fragment 重建、旋转、权限页/蓝牙设置返回均不创建自动任务。
4. 冷启动只在以下条件全部成立时创建一次自动任务：默认配置存在且合法；当前没有手动/默认任务；本次 cold-start token 尚未消费。
5. token 在决定“不触发、等待权限或已创建任务”时均视为已消费，生命周期回调不能再次触发。权限允许或蓝牙开启后只继续原挂起任务。
6. 自动流程可由用户取消；失败后只提供显式重试，不在同一次启动内循环重试。
7. 用户手动重试不受本次冷启动 token 限制，但仍受全局单任务锁。
8. 编辑正在执行任务对应的配置不改变任务快照；重试时读取最新配置。

### 状态机

```text
Idle
  → Preconditions
  → Connecting（若方案 B 则先 Discovering）
  → DiscoveringServices
  → PreparingChannel
  → ReadingSeed
  → SendingCommand
  → Sent / Failed / Cancelled / TimedOut
```

- `Sent` 只表示 GATT 写回调成功；没有可信设备业务回执时不命名为 `Opened`。
- 每个 taskId 只能到达一个终态；终态后所有回调忽略。
- 每阶段有有限超时，全流程另有上限；具体秒数待目标门禁实测后确认。

### 字段与校验规则

| 字段 | 规则 | 错误处理 |
|---|---|---|
| `id` | APP 生成、稳定、唯一、不可编辑 | 冲突时拒绝写入并记录非敏感诊断 |
| `doorName` | trim 后 1–80 字符；展示必填 | 就地提示，不保存 |
| `mac` | 接受用户常见大小写输入，保存为大写冒号格式；须通过系统地址校验 | 就地提示示例格式，不连接 |
| `key` | 推荐保存为恰好 16 位大写 hex；不允许奇数、非 hex、静默补零或截断 | 密码输入控件；默认隐藏；校验失败不保存 |
| `bluetoothName` | 只在保留旧扫描兼容时可选保存；不得成为 SafeBaiyun 直连前置条件 | 缺失不阻塞方案 A |
| `defaultId` | null 或现存 id；和配置列表一次事务写入 | 悬空引用读取时清空，不自动改选 |

## 异常与边界

| 场景 | 预期行为 |
|---|---|
| 存储为空/首次启动 | 空状态；无连接、无权限循环弹窗 |
| 存储解密失败/JSON 损坏 | 不崩溃、不覆盖原数据；提示配置不可读取并提供安全重置确认 |
| 重复 MAC | 保存前提示冲突并定位已有项；默认不创建重复项 |
| 删除默认门禁 | 明示同时取消默认；确认后两项原子更新 |
| 默认 id 悬空 | 清空关系，记录非敏感诊断，不自动选第一项 |
| 蓝牙不支持 | 阻止开门，给出设备不支持提示 |
| 蓝牙关闭 | 引导开启；拒绝后终止原任务，不反复触发 |
| 权限首次拒绝/永久拒绝/撤回 | 分别给再次申请或设置入口；任务终止，不崩溃 |
| 设备不可达/连接超时 | 单次失败、清理 GATT、显式重试；不无限重连 |
| Service/特征缺失 | 协议不兼容失败；不尝试未知写入 |
| seed 为空/过长/读取失败 | 协议失败；不得构造或发送帧 |
| DES 失败 | 终止且不发送空/部分帧 |
| GATT API 返回未发起或回调非成功 | 对应阶段失败；清理资源 |
| 指令写入成功 | 显示“指令已发送，请确认门禁状态” |
| 用户取消同时收到成功回调 | `Cancelled` 胜出；迟到回调忽略 |
| Activity 重建/前后台切换 | 恢复同一 taskId 视图，不创建新任务 |
| 连续点击/默认与手动竞争 | 只有首个取得锁的任务执行；其余展示进行中 |
| 流程中编辑/删除配置 | 当前任务用启动快照；删除不强制把已发送结果改写，重试读取最新状态 |
| 飞行模式 | 用户按系统允许单独开启蓝牙后，完整路径可执行且无网络依赖 |

## 非功能需求

- 安全：key 使用 Android Keystore 支持的加密存储；`allowBackup=false`；UI、Toast、日志、异常、分析事件、剪贴板和分享中不出现完整 key、seed 或完整可重放帧。
- 可靠性：全局最多一个任务；每个任务恰有一个终态；取消、超时和生命周期销毁后释放 GATT、回调与计时器。
- 性能：不阻塞主线程；单次失败在阶段超时或全流程上限内返回。具体时长由真机基线确认，不在需求阶段伪造性能承诺。
- 可用性：每个失败提供下一步（授权、开蓝牙、检查距离/配置、重试或返回列表）；无默认时不打扰。
- 隐私：不申请与确认连接方案无关的权限；采用直连时不申请扫描/定位权限。
- 可访问性：状态不只依赖颜色；按钮具备明确文本和可读焦点；进行中按钮防连点但保留取消入口。
- 可测试性：协议纯函数有绑定 SafeBaiyun commit 的黄金向量；调度器、存储、迁移和错误映射可在无真实 key 的替身环境测试。
- 兼容性：Android API 26–35；目标门禁与各主版本真机覆盖见兼容矩阵。
- 可维护性：业务状态机和 Activity/Fragment 分离；协议基线、扩展能力和 UI 字段不得混为一体。

## 数据、权限与合规

- 数据只在设备本地使用，不上传后端。门禁 key 属高敏感物理访问凭据。
- 最小数据集合：稳定 id、显示名、MAC、加密 key、可选兼容字段；默认关系单独保存为 `defaultId`。
- APP 不提供从第三方应用数据库提取 key 的工具或教程；用户必须对录入凭据拥有授权。
- 采用直连方案时，API 26–30 使用 legacy Bluetooth manifest 权限，API 31+ 运行时请求 `BLUETOOTH_CONNECT`；不申请扫描/定位。
- 采用扫描方案时才按矩阵增加 `BLUETOOTH_SCAN` 和旧版本定位权限，并解释用途。
- 删除配置时同时删除对应加密数据和默认引用；不承诺安全擦除闪存物理副本。
- 崩溃报告与遥测若后续接入，必须建立敏感字段拦截；本需求不新增遥测。

## 依赖与约束

- 指定协议参考 commit 必须可访问并固定；后续上游变更不自动进入需求。
- 需要至少一套授权测试门禁、测试 MAC/key 和可观察物理结果的环境；证据不得进入仓库或 Issue。
- 需要用户确认三份 ADR 和需求 commit 后才能进入开发。
- 当前仓库没有 Android 工程；开发阶段必须从用户确认的需求 commit 新建可编译基线。
- 当前 CI/CD 未接入，所有构建、测试和真机证据须如实记录为手动执行。
- DES 与设备协议属于遗留约束；APP 侧加密存储不能弥补链路和设备端密码学缺陷。

## 发布与度量

本 PRD 不授权发布。后续阶段必须绑定确认需求 commit、开发 PR head SHA、测试 build ID、QA PASS、用户 UAT 和生产批准。

候选度量只用于 QA/本地验收，不新增联网遥测：

- CRUD、默认唯一性和重启恢复用例通过率 100%。
- 每次冷启动自动任务创建数为 0 或 1；并发 GATT 任务最大值为 1。
- 所有失败注入在有限时间内到达唯一终态且资源释放。
- 飞行模式下手动和默认路径覆盖通过。
- 日志与导出产物敏感信息扫描 0 泄漏。
- 各 Android 主版本和目标门禁兼容矩阵记录完整。
