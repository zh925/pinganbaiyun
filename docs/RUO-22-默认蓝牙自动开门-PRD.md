# RUO-22 默认蓝牙自动开门 PRD（定稿）

> 归档日期：2026-07-16。需求与原型已由需求设计小队确认；对应原型为 `default-bluetooth-prototype.html`。

## 1. 目标与范围

用户可从“蓝牙信息”中配置唯一默认门禁。APP 冷启动、热启动或从后台回到前台时，在满足条件且未命中重复抑制规则的情况下，自动复用现有列表点击开门链路。

本需求不修改 BLE/门禁协议，不新增云端鉴权。协议能力未确认前，成功态统一使用“开门指令已执行”等中性文案，不承诺物理门已打开。

## 2. 已确认规则

1. 默认设备最多一个，也允许为空；支持设置、更换、取消，删除默认设备时同步清空且不自动改选。
2. 默认关系以配置稳定 `id` 对应，独立持久化为 `defaultId`，不得复用会随新增/编辑变化的 `currentId`。
3. 冷启动、热启动、后台到前台均执行检查；Activity/Fragment 重建、权限页或系统弹窗返回不得产生重复任务。
4. 自动开门不二次确认，连接、超时、错误分类和重试严格复用 `BleUnlockManager` 及列表点击开门现状。
5. 同时仅允许一个开门任务；成功后同设备 30 秒内静默抑制生命周期再次触发。用户主动点击/重试不受 30 秒限制，但仍受单任务锁限制。
6. 自动失败使用 Toast 提示，并在可见页面保留“重试”；重试使用默认设备的最新配置，不自动切换其他设备。
7. 有效 NFC 目标优先且不并发默认任务。NFC 无效、未绑定或失效且有有效默认设备时，询问“使用默认设备 / 取消”；无默认时沿用 NFC 失败流程。
8. 用户未设置默认设备且首次列表手动开门成功后，展示一次可跳过的弱引导。

## 3. 状态与交互

### 列表

- 默认项显示唯一“默认”标识和“取消默认”入口；非默认项显示“设为默认”。
- 已有默认时设置另一项，弹窗明确旧门禁和新门禁；确认后一次性替换。
- 删除默认项的确认文案明确“同时取消默认”。
- 首次手动开门成功弱引导可“设为默认 / 暂不”，仅展示一次。

### 自动流程

- 至少覆盖准备、扫描/连接、开门中、成功、失败。
- 蓝牙关闭和权限缺失沿用既有引导；未就绪前不得下发指令。
- 失败 Toast 展示实际原因，页面保留重试和返回门禁列表入口。
- 成功使用中性文案“开门指令已执行”，提示用户留意门禁实际状态。

### NFC 与防并发

- 有效 NFC 目标优先。
- 无效 NFC 有默认时需用户确认后才能降级使用默认设备。
- 已有任务时再次回前台或碰卡，不创建第二个任务，并展示当前任务状态/提示。

## 4. 验收要点

- 设置、更换、取消、删除后默认关系正确，重启仍保留，任何时刻最多一个默认。
- 无默认时生命周期事件不扫描、不连接；有默认时每个合法前台事件最多创建一个任务。
- 进行中任务、30 秒抑制窗、NFC 优先级均不会造成并发连接或重复下发。
- 仅 `CONNECT_FAILED` / `SCAN_TIMEOUT` 复用既有一次自动重跑；其他失败不自动重跑。
- 自动失败有 Toast 与显式重试；成功文案不写死“物理门已打开”或具体回执。
- 首次弱引导可跳过且不重复打扰。

## 5. 外部依赖

- 协议方确认 `command=04` 握手成功是否等同实际开门，以及 `00/02` 最终回执语义。
- 设备端幂等、防重放、稳定身份规则及目标门禁真机表现。
- NFC “未绑定/失效”的数据来源；当前本地实现仅能校验卡内四要素。

这些依赖不阻塞 UI 状态、默认关系和调度框架落地，但不得因此改变 BLE 协议或夸大成功语义。
