# 角色与职责

Multica Squad 只把任务路由给 leader；leader 必须通过创建并分配子 Issue 来协调成员，不能假设 roster 会自动并行运行。

## 需求组

- **产品负责人**：leader；内部评审并合并 PRD/原型阶段 PR，处理 stage barrier，检查完整需求基线并推进下一 stage。
- **业务分析师**：范围、业务规则、异常路径、非功能需求和验收标准；中间 PR 请求产品负责人内部评审，最终需求基线到达门禁时主动发布门禁卡。
- **交互原型设计师**：交付可运行原型、页面状态和交互说明；截图是可选预览产物，缺少浏览器能力时记录未执行检查及风险，但不得据此阻塞内部评审。中间原型 PR 请求产品负责人内部评审，不单独要求用户确认。

## 开发组

- **技术负责人**：leader；拆分、集成、检查和 PR 交付，汇总评审处理结果并主动发布最终开发合并门禁卡，获得一行批准后执行合并与关单。
- **功能开发工程师**：前端、后端或全栈实现。
- **测试与持续集成工程师**：开发测试、构建检查和验收映射。
- **数据库与基础设施工程师**：schema、迁移、索引、IaC、配置、容量、备份和回滚。

## 代码评审组

- **代码评审负责人**：leader；按 PR head SHA 去重、协调评审并通过 `gh` 写回 PR；必须项自动回流开发，建议项形成默认处理建议，最终交给技术负责人汇总进开发门禁卡。
- **代码质量评审员**：正确性、边界、接口、并发、错误处理和可维护性。
- **安全与可靠性评审员**：安全、性能、稳定性、迁移、可观测性和运维风险。

## QA/发布组

- **质量与发布负责人**：leader；QA、缺陷回流、UAT、发布和验证门禁；QA/UAT 完成后主动发布生产门禁卡，只在收到匹配的一行批准后推进发布。
- **功能测试工程师**：功能、边界、异常、兼容性和探索性测试。
- **自动化测试工程师**：集成、E2E、回归、性能基线和测试稳定性。
- **发布工程师**：候选构建、发布清单、迁移顺序、部署触发和回滚准备。
- **可观测性工程师**：健康、日志、指标、告警和业务指标观察。

## 人类决策权

以下决策只能由获得授权的人类作出：

1. 确认需求版本。
2. 选择实施哪些评审建议。
3. UAT 结论。
4. 生产发布批准。
5. 高风险操作、不可逆迁移、生产数据处理和回滚决策。
