# 状态与人工门禁

Multica 状态本身不强制流程，因此 leader 和参与者必须执行以下门禁。

## 状态语义

- `backlog`：前置条件未满足，不触发 Agent。
- `todo`：前置条件已满足，可以开始。
- `in_progress`：正在执行。
- `in_review`：等待同阶段评审或人工决策。
- `blocked`：外部条件阻塞，必须写明 `waiting_on` 和原因。
- `done`：阶段产物和证据完整，且所需门禁已通过。
- `cancelled`：明确取消，保留历史。

## 用户只确认三个关键节点

1. **最终需求基线**：PRD、原型和内部一致性评审全部完成后确认一次。
2. **开发 PR 最终合并**：必须修改项闭环、建议项已形成处理结论后确认一次。
3. **生产发布**：QA PASS、指定版本 UAT 通过、发布与回滚计划完整后确认一次。

PRD、原型等中间阶段由需求组内部评审，不要求用户逐阶段确认。高风险数据库迁移、生产数据、不可逆操作和回滚另行使用完整审批，不合并进普通门禁。

## 一行门禁口令

Agent 到达关键节点时必须主动发布预填充门禁卡：

```text
门禁编号：GATE-RUO-123-a1b2c3d
门禁类型：最终需求基线 / 开发 PR 最终合并 / 生产发布
确认对象：PR URL 或发布对象
确认版本：完整 commit SHA / build ID
检查摘要：
评审与遗留风险：
拟执行动作：
默认策略：squash / 发布计划

请回复其一：
批准 GATE-RUO-123-a1b2c3d
拒绝 GATE-RUO-123-a1b2c3d：原因
```

用户不需要重复粘贴 PR URL 和完整 SHA。Agent 必须从被回复或同一线程中的门禁卡读取上下文，并在执行前重新核验。

## 评审建议处理

```text
必须修改：由开发自动闭环，未闭环不能发布开发合并门禁卡
建议修改：评审负责人给出默认“采纳/延期”建议并在门禁卡汇总
产品或范围取舍：作为遗留风险列入开发合并门禁卡，由用户一次确认
```

## 高风险完整审批

```text
操作对象：
精确版本：
影响范围：
执行窗口：
备份与回滚：
批准结论：通过 / 拒绝
批准人：
```

高风险审批缺少必要字段时，Agent 必须继续等待，不得使用普通一行口令替代。

## 评论后的强制动作

人工批准不是信息归档事件。当前交付 Issue 的 assignee 收到一行口令后必须：

1. 核对评论作者是有决策权的人类成员。
2. 找到唯一、未失效且编号完全一致的门禁卡；找不到或存在多个候选时停止并重新发卡。
3. 重新读取 Issue、关联 PR、当前 head SHA 或 build ID；与门禁卡不一致时旧卡立即失效并重新发卡。
4. 确认 PR Ready、可合并，并如实报告 checks；CI/CD 未接入时不得伪造通过结果。
5. 收到 `批准 <门禁编号>` 后，确认 PR 正文包含 `Closes <当前交付 Issue>`，再执行门禁卡中的动作。
6. 只有确认 GitHub PR 状态为 `merged` 后，才核验或设置当前 Issue 为 `done`。
7. 将 PR URL、merged commit、执行结果和未执行项写入 Issue 评论。

收到 `拒绝 <门禁编号>：原因` 时，Issue 保持 `in_review`，记录原因并回到相应修改阶段。

完整状态机见 [批准执行与阶段推进](approval-execution.md)。
