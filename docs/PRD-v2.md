## 一、背景与目标

「平安白云」是一款安卓原生 APP。整体链路是一个**门禁开门闭环**：

- **配置流程**：在 APP 内维护门禁参数 → 选门 → 贴卡把参数写入 NFC 卡 → 该卡成为「门禁卡」。
- **运行流程**：用户碰卡 → 系统冷启动唤起 APP → 读出卡内门禁参数 → 自动按 MAC 连接对应 **BLE** 门禁 → **DES 握手 → 开门 → 解析回执**。

**本期目标**：搭建可运行的工程框架并跑通上述闭环（P0），数据全本地、无后端。

**成功定义**：拉取工程可 Gradle 构建、安装运行；具备 NFC（读+写+冷启动）/ 蓝牙（BLE 扫描连接 + DES 开门）/ 蓝牙信息维护 / 基础设施分层与导航骨架；能以对照参考仓库校验过的协议实现，走通「维护参数 → 写卡 → 碰卡冷启动 → 读参数 → 连接 → 握手 → 开门」全链路。

## 二、范围

- **纳入（本期 P0）**：工程骨架；NFC 读卡 + **写卡** + 冷启动唤起；BLE 扫描/连接/DES 握手开门/回执解析；蓝牙信息维护页 CRUD + 本地加密存储；权限管理（免定位）；导航；统一错误与状态展示。
- **暂不纳入（后续迭代）**：一卡多门、NDEF 载荷二次加密、配置导入导出、维护页生物识别锁、多设备并发。

## 四、核心协议（源自参考仓库 BaiyunKeys-preview，已逐文件核对）

- **蓝牙类型**：BLE / GATT。
- **Service UUID 候选**（命中其一即可）：`0734594a-a8e7-4b1a-a6b1-cd5243059a57`、`14839ac4-7d7e-415c-9a42-167340cf2339`。特征按 notify/read/write 属性**动态选**，不写死。
- **加密**：单 DES（ECB / NoPadding，8 字节块；CryptoJS.DES 只取 key 前 8 字节，**非 3DES**）。
- **帧格式**：`A5 <len> … <checksum> 5A`，`checksum = (~(sum&0xff)+256)&0xff`，置倒数第二字节。
- **指令构造（安卓需 1:1 移植）**：`buildHandshakeCommandWithHeader` / `buildCommKeyCommand` / `buildTimeSyncCommand`(`generateTimeHex` BCD 时间) / `encryptUnlockCommand` / `decodeOpenResult`。帧头 = MAC[2:6] 4 字节 + 蓝牙名解析出的设备标识。
- **回执**：解密后状态字节 `00`=开门成功、`02`=门已打开、其它=开门失败（密码无效）。
- **超时**：扫描 10s / 连接单步 8s / 读 seed 5s / 用户 ACK 3.5s / 全流程 45s。
- **开发校验**：用仓库同一 (key, seed) 造黄金用例，确保安卓端 DES 加解密与帧构造逐字节一致。

详见附件《蓝牙门禁协议规格.md》（上一评论）。

## 五、NFC 数据模型与写卡

**卡内 / 本地门禁配置 = 四要素**（字段与校验取自 `config.js`）：

| 字段 | 含义 | 校验 |
|------|------|------|
| `doorName` | 门/设备显示名 | 非空文本，trim |
| `mac` | 蓝牙 MAC | `^([0-9A-F]{2}:){5}[0-9A-F]{2}$`，大写 |
| `key` | 开锁密钥(DES key, hex) | `^[0-9A-F]+$`，长度 16–32 且偶数，大写 |
| `bluetoothName` | 蓝牙广播名（解析 deviceId 头部） | 大写字母数字，长度足够（≥11，开发阶段对 `extractDeviceIdParts` 取值区间核对固化） |

- **载荷格式**：单条配置序列化为 JSON，以**自定义 MIME 类型 `application/vnd.pinganbaiyun.door`** 的 NDEF Record 写入卡（本期一卡一门）。
- **冷启动**：Activity 声明 `NDEF_DISCOVERED` + 该 MIME 的 intent-filter → APP 未启动时被精确冷启动唤起（不弹选择框），实现「碰一碰」；前台运行时用 `enableForegroundDispatch`/ReaderMode + `onNewIntent` 统一入口。
- **写卡流程**：维护页选门 → 贴卡 → 组装 MIME NDEF Message 写入 → **写后回读校验四要素一致** → 成功/失败可重试。异常分支：只读卡、容量不足、中途移开、非本 APP 数据（二次确认后覆盖）。

## 六、功能列表与优先级

**NFC 模块**
- P0 NDEF 四要素编解码；自定义 MIME `application/vnd.pinganbaiyun.door` 的冷启动 intent-filter
- P0 前台调度 / ReaderMode 读卡 + `onNewIntent` 统一入口
- P0 **写卡**（组装 MIME NDEF + 写后回读校验）
- P1 NFC 不支持/未开启检测与引导；写卡覆盖二次确认
- P1 「清卡」入口

**蓝牙模块（BLE）**
- P0 运行时权限（`BLUETOOTH_SCAN` neverForLocation / `BLUETOOTH_CONNECT`，免定位）+ 蓝牙开关检测引导
- P0 Service UUID 候选过滤扫描 → GATT 连接 → 服务/特征动态选 → 开启 notify
- P0 DES 握手 → 密钥/时间同步 → 开门指令 → 回执解析（状态机集中管理）
- P0 开门结果状态（成功 00 / 门已开 02 / 失败）
- P1 断线检测与重连/重试；扫描超时自动重扫

**蓝牙信息维护**
- P0 设备列表（门名/蓝牙名/MAC，key 掩码）
- P0 新增/编辑表单（四要素实时校验）+ 删除（二次确认）
- P0 每条「写入 NFC 卡」入口直连写卡
- P0 本地持久化 + **key 加密存储**（EncryptedSharedPreferences / Keystore）
- P2 导入/导出、默认门置顶

**基础设施**
- P0 工程骨架：项目结构 / Gradle / 依赖 / 分层（MVVM）
- P0 权限统一管理、页面导航骨架、统一错误处理 + 状态展示
- P1 本地日志（掩码 key）
- —（无后端，不做网络层）

## 七、用户故事

US-1 碰卡唤起 · US-2 读卡反馈 · US-3 自动连接（→握手开门）· US-5 状态可见（→开门成功/门已开/失败）· US-6 权限引导 · US-7 异常恢复（承第一轮）
- **US-8 维护门禁参数**：管理员对门禁配置增删改查，管理多把门。
- **US-9 写卡**：管理员把某门禁配置写入 NFC 卡，让持卡人碰卡即用。
- **US-10 碰卡开门**：使用者碰卡后 APP 自动读参数、连对应门禁并开门，无需手动选设备。
- **US-11 冷启动唤起**：APP 未启动时碰卡也能被拉起并直接进入开门流程。

## 八、非功能需求

- **安全**：① `key` 明文写入 NDEF 卡属**固有风险**（任何读卡器可读出开锁密钥）——本期决策接受明文 + 本地加密存储（见九·缺口2，待提出方最终确认）；② 本地 `key` 用 EncryptedSharedPreferences/Keystore 加密；③ 日志/UI 一律掩码 key。
- **兼容 · 卡容量**：四要素 JSON（含 32 hex key）体积偏大，**NTAG213(144B) 可能不足**，建议目标卡用 **NTAG215/216**；写卡前检测容量并给「容量不足」提示。
- **权限**：Android 12+ 声明 `BLUETOOTH_SCAN`(neverForLocation)/`BLUETOOTH_CONNECT`，**不申请定位**；NFC 权限 + `NfcAdapter` 可用性检测。
- **性能**：沿用协议超时；碰卡到开门成功在全流程超时(45s)内完成，正常场景数秒级。
- **可靠性**：失败/超时可重试；进程被回收后再次碰卡能恢复流程；扫描/连接/加解密不阻塞主线程。
- **可维护性**：NFC/蓝牙/维护/基础设施分层解耦；连接与开门状态机集中管理。
- **可测试性**：DES/帧/指令构造/回执解析可单测；含 DES 黄金用例（与仓库逐字节一致）。

## 九、验收标准（Given-When-Then）

- **AC-1 碰卡冷启动**：APP 未启动、NFC 开启，碰本 APP 格式 NDEF 卡 → 系统按 MIME intent-filter 冷启动唤起（不弹选择框）并读出四要素；前台时前台调度稳定读卡。
- **AC-2 读卡结果**：识别成功显示四要素并进入自动连接；卡类型不符/读取失败给明确提示可重试，不崩溃。
- **AC-3 自动连接**：读卡成功且权限/开关就绪，≤2s 自动开始按 Service UUID 过滤扫描并连 MAC 对应门禁，无需手动选。
- **AC-4 扫描与选择**：发现命中设备则连接或列表展示；扫描超时(10s)提示「未找到设备」可重试。
- **AC-5 连接·握手·开门**：依次呈现 连接·握手 → 开门中 → 结果；回执 `00`→开门成功、`02`→门已打开、其它→开门失败（密码无效），文案与状态字节一致；任一步超时判失败给重试。
- **AC-6 权限与开关引导（免定位）**：Android 12+ 仅申请 `BLUETOOTH_SCAN`(neverForLocation)/`BLUETOOTH_CONNECT`，不申请定位；缺权限/开关给明确引导，不静默失败。
- **AC-7 异常恢复**：任一环失败，点重试从合理节点重新执行。
- **AC-8 工程框架**：拉取工程 Gradle 构建编译通过、可安装运行，具备分层结构与导航骨架。
- **AC-9 写卡**：维护页某条配置有效，点「写入 NFC 卡」贴可写卡 → 以 MIME 写入四要素、回读校验一致、显示成功；只读/容量不足/中途移开显示对应失败可重试，不崩溃。
- **AC-10 维护页 CRUD**：新增/编辑时 MAC 与 key 实时校验、非法拦截；删除并确认后本地列表移除且持久化。
- **AC-11 本地存储与安全**：重启 APP 列表完整恢复；展示 key 默认掩码、日志无明文 key。
- **AC-12 DES 黄金用例**：仓库同一 (key, seed)，安卓端 DES 加解密与帧构造输出与小程序端逐字节一致。

## 十一、本期决策与开放项

**已落定（提出方答复 + PM 拍定）**：Q1 NDEF + 支持写卡；Q2 BLE 门禁（参考仓库，连后=开门）；Q3 数据全本地；Q4 较新 Android + 免定位；Q5 无后端无账号；Q6 NDEF 冷启动碰一碰；MIME=`application/vnd.pinganbaiyun.door`；单 DES；一卡一门；key 本地加密+掩码；写卡覆盖=空白卡直写、非本 APP 数据二次确认覆盖、清卡入口 P1；写卡纳入本期 P0。

**唯一开放项（缺口 2，待提出方最终确认，不阻塞落地）**：
1. **目标 NFC 卡型号** —— 建议统一 NTAG215/216（NTAG213 容量可能不足）。
2. **key 明文入卡安全取舍** —— 建议本期接受明文（快速跑通闭环）+ 本地加密存储，NDEF 载荷二次加密作为后续增强（全本地无后端下二次加密仅混淆级，防不住逆向）。

> 提出方如对以上默认方案有不同意见，回复后按增量微调；无异议即以本 v2 为准进入框架落地。
