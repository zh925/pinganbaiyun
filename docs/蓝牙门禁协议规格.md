# 蓝牙门禁协议规格（源自参考仓库 BaiyunKeys-preview 逆向提炼）

> 来源：提出方在 Q2 提供的 `github.com/1136623363/BaiyunKeys-preview`（"基于平安回家 App 复刻的蓝牙门禁微信小程序"）。
> 本规格是把小程序端的 BLE 门禁协议提炼为安卓可移植的规格，供需求/开发对齐。关键源文件：`miniprogram/utils/bleProtocol.js`、`utils/lockBiz.js`、`utils/des.js`、`utils/config.js`、`pages/index/index.js`。

---

## 0. 一句话链路

NFC 卡里存**门禁设备参数** → 碰卡冷启动 APP → 读出参数 → 按 Service UUID 扫描并连接对应 **BLE** 门禁 → **DES 加密握手 → 开门指令 → 解析回执**。
> 印证：Q1「从 APP 为 NFC 卡写入数据」= 把某个门禁配置写进卡；Q2「维护蓝牙信息的页面」= 设备参数增删改查页，写卡入口接在其中；Q3「数据全本地」= 参数存本地。

## 1. 蓝牙类型

**BLE（低功耗）**。小程序用 `wx.startBluetoothDevicesDiscovery / getBLEDeviceServices / getBLEDeviceCharacteristics / notifyBLECharacteristicValueChange / onBLECharacteristicValueChange / read/writeBLECharacteristicValue`，对应安卓 **GATT**。

## 2. 扫描与连接流程（对应 `pages/index/index.js`）

1. **扫描**：`startBluetoothDevicesDiscovery` 带 `services = SERVICE_CANDIDATES` 过滤，`allowDuplicatesKey:false`；发现超时 `DISCOVERY_TIMEOUT = 10000ms`。
2. **连接**：连 GATT；成功后 `getBLEDeviceServices` 找到匹配的目标 service。
3. **特征通道**（`prepareChannel`）：在目标 service 下**按属性动态选**特征——`notify` 属性→notifyId，`read`→readId，`write`→writeId（非硬编码 UUID）。
4. **开启通知** → `onBLECharacteristicValueChange` 监听回执。
5. **读随机数种子 seed**（readSeed，`READ_TIMEOUT = 5000ms`）。
6. 用 seed + 设备 key 构造**握手指令**并写入 writeId。
7. 等门锁 ACK（`USER_ACK_TIMEOUT = 3500ms`），后续通信密钥/时间同步/开门指令走同一 write/notify 通道。
8. 全流程超时 `FLOW_TIMEOUT = 45000ms`；扫描超时支持自动重发。

**Service UUID 候选**（命中其一即可）：
- `0734594a-a8e7-4b1a-a6b1-cd5243059a57`
- `14839ac4-7d7e-415c-9a42-167340cf2339`

## 3. 设备参数（= NFC 卡承载的数据，`utils/config.js` / `lockBiz.js`）

每个门禁一条配置，四要素：

| 字段 | 说明 | 校验 |
|------|------|------|
| `doorName` | 门/设备显示名 | 文本 |
| `mac` | 蓝牙 MAC 地址 | `^([0-9A-F]{2}:){5}[0-9A-F]{2}$` |
| `key` | 开锁密钥（DES 密钥，hex） | hex，长度 16–32 且为偶数（8–16 字节） |
| `bluetoothName` | 蓝牙广播名，用于解析 deviceId 头部 | 大写字母数字，长度 ≥ 11 |

> `extractDeviceIdParts(bluetoothName)` 从蓝牙名第 3~10 位解析出 4 字节设备标识，用于指令帧头部；`deriveHeaderFromMac(mac)` 取 MAC 第 2~6 字节。二者在指令构造中都会用到。

## 4. 加密与指令帧（`utils/bleProtocol.js` / `des.js` / `lockBiz.js`）

- **加密算法**：**DES**（ECB 单块，`desEncryptBlockHex/desDecryptBlockHex`）。key 为设备 `key`。
- **帧格式**：`A5 <len> ... <checksum> 5A`，`checksum = 帧字节和的补码 (~sum & 0xff)`，倒数第二字节。
- **主要指令构造函数**（安卓需 1:1 移植）：
  - `buildHandshakeCommandWithHeader(seed, headerBytes, key)` — 握手（帧头 `A5 14 05 ...`）
  - `buildCommKeyCommand(...)` — 通信密钥协商（`A5 16 01 ...`，body 含 sum+xor 校验）
  - `buildTimeSyncCommand(timeBytes, ...)` — 时间同步（`generateTimeHex` 生成 BCD 年月日时分秒周）
  - `encryptUnlockCommand(seed, mac, key)`（`lockBiz.js`）— 开锁指令
- **回执解析** `decodeOpenResult(frameHex, key)`：DES 解密后取状态字节——`00`=开门成功、`02`=门已打开、其它=失败（密码无效）。

## 5. 对安卓框架的映射结论

- 蓝牙模块按 **BLE/GATT** 设计；扫描过滤用 Service UUID 候选表；特征按属性动态选，不写死。
- 需实现 **DES(ECB) + A5..5A 帧编解码 + 上述 4 类指令构造 + 回执解析**（可直接对照仓库源文件逐函数移植并写单测）。
- **NFC 写卡数据模型 = 上述四要素**（doorName/mac/key/bluetoothName）。NDEF 载荷格式由本 APP 自定（建议自定义 MIME type + JSON，冷启动 intent-filter 匹配该 MIME）。
- 数据全本地（Q3/Q5）：设备参数与卡数据存本地，无后端、无账号。

## 6. 待进一步核对（开发阶段）

- 通信密钥协商→时间同步→开门 的**完整时序与各指令触发条件**，以仓库 `pages/index/index.js` 主流程 + notify 回调为准逐步还原。
- `des.js` 是标准 DES 还是变体（密钥调度/填充），需比对确保安卓端加解密结果一致（用仓库同一 key/seed 做黄金用例）。
- key 长度 16 vs 32 时是否切换 3DES，需实测确认。
