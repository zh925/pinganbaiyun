## 4. 核心设计决策（本期锁定）

1. **加密**：单 DES / ECB / NoPadding，8 字节块；密钥清洗为大写 hex 后**只取前 8 字节**（前 16 hex），**非 3DES**。JCE `"DES/ECB/NoPadding"` 与 CryptoJS.DES 输出一致（已用 FIPS 官方向量 `133457799BBCDFF1 / 0123456789ABCDEF → 85E813540F0AB405` 核对）。
2. **帧格式**：`A5 <len> … <checksum> 5A`，`checksum = (~(帧字节和 & 0xff)) & 0xff`，置倒数第二字节。
3. **BLE**：BLE/GATT；**主路径为已知 MAC 直连**（对齐 SafeBaiyun：`checkBluetoothAddress` 校验后 `getRemoteDevice(mac).connectGatt(TRANSPORT_LE)`，不扫描）——门锁广播不被 Android `startScan` 收到，但 GATT 可向已知 MAC 直连。直连失败（连接期）转一次无过滤扫描兜底（靠 5 重匹配定位门锁）。候选 Service UUID 表用于连接后「选服务」。特征**按属性动态选** notify/read/write，不写死 UUID。超时：扫描 10s / 连接 8s / 读种子 5s / ACK 3.5s / 全流程 45s。
4. **NFC**：四要素序列化为 JSON，以自定义 MIME `application/vnd.pinganbaiyun.door` 的 NDEF Record 承载；同一 MIME 的 `NDEF_DISCOVERED` intent-filter 实现冷启动精确唤起（不弹选择框）。一卡一门。
5. **权限**：Android 12+ 声明 `BLUETOOTH_SCAN`(neverForLocation) / `BLUETOOTH_CONNECT`、不申请定位；**Android 8～11（minSdk=26 覆盖）BLE 扫描必须有 `ACCESS_FINE_LOCATION` 且定位开关开启**，故 manifest 声明 `ACCESS_FINE_LOCATION`(`maxSdkVersion=30`)、运行时对 `<31` 设备申请该权限并在定位开关未开时给可读提示。NFC 权限 + `NfcAdapter` 可用性检测。
6. **数据全本地**：无后端、无账号。四要素（含明文 key）落盘一律经 EncryptedSharedPreferences（AES256-GCM，主密钥入 Keystore）。UI/日志一律用 `DoorConfig.maskedKey` 掩码。

## 5. 预留给 UI 层（阶段 2）的接口约定

- **读卡**：`NfcController(activity)`；`onResume→enableForegroundDispatch()`、`onPause→disableForegroundDispatch()`；`onCreate/onNewIntent` 的 Intent 交 `readFromIntent(intent): DoorConfig?`。可用性：`isSupported` / `isEnabled`。
- **写卡**：贴卡拿到 `Tag` 后 `writeConfig(tag, config): NfcWriteResult`（`Success` / `Failure(error, message)`，error∈{READ_ONLY, CAPACITY, NOT_NDEF, VERIFY, IO}，已含写后回读校验与容量检测）。`extractTag(intent)` 取 Tag。写前容量预估：`DoorNdef.encodedByteLength(config)`。
- **开门**：`BleUnlockManager(context).apply{ listener=... }.start(config)`；订阅 `UnlockStateListener` 收 `UnlockState`（Scanning/Connecting/PreparingChannel/ReadingSeed/Opening/Success(result)/Failed(reason,error)），全部在主线程回调。`stop()` 释放资源。**调用前须先申请好蓝牙权限**（引擎层不做权限弹窗）。
- **维护/存储**：`DoorConfigStore.get(context)`；`list()` / `get(id)` / `upsert(config): DoorConfig` / `delete(id)` / `setCurrent(id)` / `currentId()`。
- **模型/校验**：`DoorConfig`（`doorName/mac/key/bluetoothName`）；`validate(): List<String>`（返回失败原因，实时校验用）/ `isValid`；`maskedKey`；`DoorConfig.normalized(...)` 规范化输入；`toJson()/fromJson()`。
- **协议纯函数**（需要时直用）：`DoorProtocol` 的 `buildHandshakeCommandWithHeader / encryptUnlockCommand / buildTimeSyncCommand / generateTimeHex / decodeOpenResult / deriveHeaderFromMac / extractDeviceIdParts / isValidMac / isValidKey`。

### 8.3 关键接线与决策

- **运行流程闭环**：碰卡 → `NfcRouterActivity.readFromIntent` 得 `DoorConfig` → `UnlockActivity`（N-03）→ 校验蓝牙权限/开关 → `UnlockViewModel.start(config)` → 状态机驱动 C 段 → 回执 00/02 成功、其它失败可重试。
- **配置流程闭环**：`DeviceEditActivity` 校验后 `DoorConfigStore.upsert` → 「保存并写入」跳 `WriteCardActivity` → `NfcController.writeConfig`（含写后回读）→ W-03/W-04。
- **权限（免定位）**：仅 Android 12+ 运行时申请 `BLUETOOTH_SCAN`/`CONNECT`；写卡与读卡在 Activity 前台调度层处理，引擎层不做权限弹窗（与 §5 约定一致）。
- **状态色系**：NFC 青 / 蓝牙蓝 / 开门·配置紫 / 成功绿 / 提示橙 / 错误红，均落 `colors.xml` 与 `bg_*` shape drawable，按语义复用。
- **`key` 脱敏**：读卡成功卡片展示一律用 `DoorConfig.maskedKey`。

## 9. 阶段 3（RUO-8）：删手动扫描页 + 蓝牙信息页每条一键开门

依据附件《RUO-8-需求文档-v2精简.md》（终审通过）与原型 `pingan-baiyun-prototype.html` 第 05 节。纯客户端交互层小改，**复用现有 BLE/DES/NFC 底层与四要素数据模型不变**，未改动 `core/` 引擎与协议层。

### 9.1 删除手动扫描门禁（需求 ①）

- 删除 `ui/scan/ScanActivity.kt`、`ui/scan/ScanAdapter.kt`、`core/ble/BleScanner.kt` 及布局 `activity_scan.xml` / `item_scan_device.xml`，移除 `AndroidManifest` 中 `ScanActivity` 声明与 `strings.xml` 的 `scan_*` / `home_scan_entry`。
- `HomeFragment` / `fragment_home.xml`：移除「手动扫描门禁」入口按钮 `btnManualScan`。
- `UnlockActivity.renderFailed()`：原 C-05「选择其他门禁」跳 `ScanActivity` 改为「返回门禁列表」——`NfcRouterActivity.devicesIntent()`（`CLEAR_TOP|SINGLE_TOP` + `EXTRA_SELECT_TAB=nav_devices`）回到蓝牙信息页，不再跳扫描。`NfcRouterActivity` 新增 `applySelectTab()` 处理该 Tab 跳转。

> §3/§8.2 中 `scan/`、`BleScanner`、B-01/02/03 屏均已随此需求删除，宿主不再有任何设备扫描列表入口。

### 9.2 蓝牙信息页每条「开门」（需求 ②）

- `item_device.xml`：设备行按钮由「写卡 / 编辑」两枚扩为「**开门** / 写卡 / 编辑」三枚 `AppCompatButton` 同档等宽（各 `weight=1`），「开门」以 `bg_button_door` 紫填充为主操作；按钮行下方新增行内反馈条 `deviceHint`（默认 `gone`，成功/失败时显示）。新增 `bg_button_ok`（成功绿）/`bg_button_err`（失败浅红描边）两个 drawable。
- `DeviceAdapter`：新增 `onOpen` 回调与单行开门状态机 `OpenState{Idle/Loading/Success/Failed(reason)}`，由 `setOpenState(id,state)` 就地刷新对应行——按钮态（开门/开门中…/已开/重试）+ `deviceHint`（成功绿 / 失败红，复用 `bg_alert_ok` / `bg_alert_warn`）。
- `DevicesFragment`：持有一个 `BleUnlockManager`（**直接复用运行流程的开门引擎**，与碰卡直连同一套连接/DES 握手/回执逻辑），懒连接——点「开门」才 `start(config)`。就绪校验（蓝牙开关 / BLE 权限，复用 `PermissionUtils` + `EnableSheets` + `ActivityResult` 启动器）与 `UnlockActivity` 一致。
  - **一次仅一行开门**：`openingConfig != null` 时忽略其它行点击；开门中按钮禁用防连点；成功后按钮保持「已开」`SUCCESS_RESET_MS=1600ms`（禁用态即冷却窗口）再复位为「开门」。
  - **重试策略（AC7）**：连接层瞬断（`CONNECT_FAILED` / `SCAN_TIMEOUT`）自动重试 1 次；其它（含 `PROTOCOL_ERROR` 鉴权失败）不自动重试，转手动「重试」。
  - 成功时 `performHapticFeedback(CONFIRM)` 触觉反馈（无需 `VIBRATE` 权限）。

> 结论：本次未触及 BLE/协议层，无需后端介入。NFC 碰卡直连、写卡、编辑等既有能力不变。

### 9.4 真机不开门定位修复（运行态，非协议层）

真机不开门经对照参考 `pages/index/index.js` 定位为运行态差异，已在 `core/ble/BleUnlockManager` 修复（加密/鉴权/成功判定/帧构造均未改）：

1. **seed 读取通道对齐**：参考小程序 `readBLECharacteristicValue` 与 notify 共用 `onBLECharacteristicValueChange`，`_stage==='waitingSeed'` 时任何值变化都当种子（`handleValueChange`）。Android 二者分离（`onCharacteristicRead` / `onCharacteristicChanged`），若门锁以 notify 回吐 seed，原走 `onCharacteristicRead` 收不到 → 读种子超时。现新增 `waitingSeed` 阶段标记，`onCharacteristicChanged` 在此阶段兜底把值当 seed 走 `handleSeedRead`（对应参考 `_seedResolve`）；read 与 notify 双通道由 `waitingSeed` 防重复处理。
2. **MTU**：连接后 `requestMtu(247)` 降低长帧分包风险（未入队则直接发现服务兜底），`onMtuChanged` 再 `discoverServices`，串行化 GATT 操作。
3. **组包**：notify 按 `A5 <len> … <cs> 5A`（len=总字节数）帧边界在 `notifyBuffer` 组包，凑够完整帧再 `dispatchNotification`，处理默认 MTU 下的长帧分包。
4. **协议级日志**：各关键节点 `Log.d("BLE", ...)`——扫描/连接/MTU/服务命中/选中的 read/write/notify 特征/握手帧 hex/`onCharacteristicRead` 的 seed/每次 `onCharacteristicChanged` 原始 value 的 hex+长度/各超时触发点，供 `adb logcat -s BLE` 确诊。

> 仍需真机抓 logcat 验证 seed 经哪条通道回吐，据日志精确收尾。`assembleDebug` + 20 单测通过。

### 9.5 App 内协议日志面板（降低真机诊断门槛）

真机抓 logcat 对用户门槛高，改用 App 内日志面板（参考小程序自带日志面板）。`BleUnlockManager` 新增 `var onLog: ((String) -> Unit)?`：私有 `log(msg)` 既 `Log.d(TAG, msg)` 落 logcat，又切回主线程推给 `onLog`（与 `listener` 一致主线程回调）；原有 `Log.d/Log.w` 统一改走 `log()`。补 `onCharacteristicWrite` 回调记录握手帧写入 `status`（`GATT_SUCCESS` 或错误码），诊断「握手帧是否真正写入设备」（`onCharacteristicWrite` 仅有 3 参 `(gatt, characteristic, status)` 形式，无 `byte[] value` 重载）。

`DevicesFragment`（一键开门入口，已持有 `BleUnlockManager`）加日志区域：点「开门」即展开，实时显示带时间戳的 `onLog` 内容（最多保留 400 行），支持「复制」(剪贴板) /「分享」(`ACTION_SEND`) /「清空」，便于用户截图反馈、无需 adb。布局 `fragment_devices.xml` 的 `bleLogPanel`（`bleLogText` + `bleLogScroll`）。`assembleDebug` + 20 单测通过。

### 9.6 扫描阶段根因修复（真机日志：10s 超时零设备）

真机日志显示扫描 10s 超时、一个设备都没扫到（卡在「开门中」实为扫描 loading）。根因在扫描阶段，修两处：

1. **位置权限（Android 8～11 必需）**：minSdk=26 覆盖 Android 8～11，这些版本 BLE 扫描必须授予 `ACCESS_FINE_LOCATION` 且定位开关开着，否则扫不到任何设备。`AndroidManifest.xml` 补 `ACCESS_FINE_LOCATION`(`maxSdkVersion=30`)；`PermissionUtils` 的 `requiredBlePermissions()` 对 `<31` 设备改为返回 `ACCESS_FINE_LOCATION`（12+ 仍 `BLUETOOTH_SCAN/CONNECT`），复用上层既有 `RequestMultiplePermissions` 启动器申请；新增 `isLocationEnabled()`（`LocationManager` 任一 Provider 开）。`BleUnlockManager.start()` 对 `<12` 且定位开关未开 fail 出可读提示「需开启手机定位…」。
2. **去掉 UUID 硬件过滤**：`start()` 不再用 `ScanFilter.setServiceUuid`（Android 对 128-bit UUID 的硬件过滤易匹配不到，门锁可能把 UUID 放 Service Data 字段），改 `startScan(null, settings, scanCallback)` 无过滤扫描、靠 MAC 匹配定位门锁（候选 UUID 表仅用于连接后选服务）。
3. **诊断日志**：`start()` 打印 `Build.VERSION.SDK_INT` + 蓝牙开关 / BLE 权限 / 定位权限授予 / 定位开关状态；`scanCallback` 记录**所有**扫到的设备（mac/name/rssi/serviceUuids）后再 MAC 匹配——日志可看到门锁是否被扫到、它广播什么 UUID。`assembleDebug` + 20 单测通过。

> 真机复测：扫描修复成功（去过滤后扫到周围几十个 BLE 设备），但门锁未出现在广播中（疑似门锁未广播/被占用）。去过滤后非门锁设备逐条 `scan device …` 把日志面板刷屏（802 行几乎全是），淹没 `start/perms/timeout/MATCH` 关键行。**降噪**：`scanCallback` 对非目标 MAC 不再逐条 log，改为每 ~1.5s 汇总一条「已扫到 N 条非目标广播（去重名字：…）」；**仅命中目标 MAC 时** log 完整 `scan MATCH mac=… name=… rssi=… uuids=…`。其余扫描/连接逻辑不动。`assembleDebug` + 20 单测通过。

### 9.7 门锁识别对齐参考 5 重匹配（根因：扫描没认出门锁）

真机日志：扫到周围几十个 BLE 设备、权限全齐，但门锁 MAC/名始终没出现 → 最可能门锁把身份（MAC/名）嵌在**广播原始数据**里、对外 `device.address`/`deviceName` 不等于配置值；参考小程序 `matchDevice`（index.js:1766-1798）为此写了 5 重匹配，App 之前只做第①条。

对齐 5 重（命中任一即判为门锁，进连接）：① 广播 MAC 精确（规范化大写无分隔比较）；② 蓝牙名相同（大写）；③ 广播原始数据含配置 MAC（正序）；④ 含配置 MAC（字节反序）；⑤ 含配置蓝牙名 ASCII。

- 纯匹配逻辑抽到 `core/ble/DoorMatcher.kt`（无 Android 依赖，可单测）：`normalizeMacHex` / `reverseMacHex`（对应 `normalizeMacForCompare` / `reverseMacHex`）/ `asciiToHex`（对应 `asciiToHex`）/ `matchWay(...)`。
- `BleUnlockManager.scanCallback` 改用 `matchDoor` → `DoorMatcher.matchWay`；命中 log `scan MATCH mac=… name=… rssi=… 命中方式=… uuids=…`（命中方式 ∈ MAC/蓝牙名/广播含MAC/广播含反序MAC/广播含名），便于复核。
- `buildAdvHex(scanRecord)` 拼可检索广播 hex（对齐小程序 `advertisData`）：原始包 `scanRecord.bytes` + 各厂商数据 value + 各服务数据 value + 服务 UUID + 设备名 ASCII——门锁身份常在 manufacturer/service data 里。
- 降噪汇总逻辑保留（非目标设备仍每 ~1.5s 汇总）。
- 新增 `DoorMatcherTest`（8 例）锁字节级对齐：MAC 规范化/反序、名 ASCII、5 重命中、无关设备不命中。`assembleDebug` + **28 单测全绿**（原 20 + 新 8）。

### 9.8 改直连模式（真根因：门锁广播不被 startScan 收到，SafeBaiyun 直连能开）

对比 `dogproton/SafeBaiyun`：它**不扫描**，`BluetoothAdapter.checkBluetoothAddress(mac)` 校验后 `adapter.getRemoteDevice(mac).connectGatt(TRANSPORT_LE)` **直接用已知 MAC 连**。BLE「扫描(scan)」依赖广播包被 `BluetoothLeScanner` 收到，而「连接(connect)」向**已知 MAC** 发起、**不依赖扫描**。这门锁广播大概率是 BLE 5 扩展广播（extended advertising），`startScan` 默认只收 legacy，故调试软件搜得到、我们搜不到、SafeBaiyun 直连能开。协议层两边字节级一致（差异 100% 在连接方式）。

改动（`BleUnlockManager`）：
- `start()` 改**直连模式**：校验 MAC → `adapter.getRemoteDevice(config.mac)` → `connectGatt(TRANSPORT_LE)` 直连，`emit(Connecting)` 进连接流程。**去掉扫描入口、去掉 `requestMtu`/`onMtuChanged`**（SafeBaiyun 无 MTU 能开门，且消除 onMtuChanged 不回调卡 45s 的隐患）；`onConnectionStateChange(CONNECTED)` 直接 `discoverServices()`。
- **连接后逻辑不动**：发现服务 → 选 `14839ac4`/`0734594a` → seed 双通道 → 握手 → 组包 → 日志。
- **扫描降为兜底**：直连在连接期失败（`onConnectionStateChange(DISCONNECTED)` 或连接超时，且尚未进入服务发现阶段）转一次扫描兜底（`startScanFallback`，沿用 5 重匹配 [DoorMatcher] + 降噪）；进入服务发现阶段（`reachedServices`）后或兜底已试过则不再回退。
- 保留 connectTimeout / flowTimeout / 全程日志。新增字段 `reachedServices`、`fallbackScanTried`。`assembleDebug` + 28 单测全绿。

> 关键：只要 `config.mac` 是门锁真实可连接 MAC（SafeBaiyun 用它直连验证过），直连即应连上、走通握手开门。

### 9.9 协议日志面板 release 门控（RUO-9 评审第 1 项·UI 层）
